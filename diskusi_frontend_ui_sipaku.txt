MASALAH TEKNIS YANG DITEMUI & SOLUSI:
====================================

1. TAILWIND TIDAK BERJALAN:
   Masalah: 
   - CSS path menggunakan absolute path
   - Classes baru belum di-build oleh Tailwind
   
   Solusi:
   - Ganti path CSS ke <?= base_url('assets/css/main.css') ?>
   - Jalankan npm run build untuk regenerate CSS
   - Tailwind v4 hanya generate classes yang benar-benar dipakai

2. VITE WARNING:
   Masalah: 
   - Warning "public directory feature may not work correctly"
   - Konflik antara outDir dan publicDir yang sama
   
   Solusi:
   - Tambahkan publicDir: false di vite.config.js
   - Disable Vite's public directory feature

PENJELASAN FOLDER RESOURCES
===========================

FUNGSI:
- Source files untuk development (CSS/JS mentah)
- Tempat Tailwind dimulai dengan @import "tailwindcss"
- File mudah dibaca, ada comments, terstruktur

WORKFLOW:
resources/css/app.css ‚Üí [Vite + Tailwind] ‚Üí public/assets/css/main.css
resources/js/main.js ‚Üí [Vite + Bundling] ‚Üí public/assets/js/main.js

PROSES TAILWIND:
1. Starting point: resources/css/app.css dengan @import "tailwindcss"
2. Scanning: @source paths untuk mencari classes yang digunakan
3. Generation: Generate CSS hanya untuk classes yang ditemukan
4. Build: Vite compile ke public/assets/

KEUNTUNGAN SISTEM INI:
- Development experience yang baik
- Hot reload saat development
- File production di-minify dan optimized
- Tree-shaking (hanya CSS yang dipakai)
- Browser caching optimal

STRUKTUR FILE YANG DIBUAT
=========================
- app/Views/auth/login.php (Halaman login lengkap)
- vite.config.js (Konfigurasi build yang dioptimasi)
- public/assets/css/main.css (CSS hasil build)

DISKUSI LANJUTAN - MODAL VS HALAMAN TERPISAH
=====================================
Tanggal: 30 Oktober 2024
Topik: Strategi Implementasi Form CRUD untuk Sistem SIPAKU

PERTANYAAN UTAMA
================
Mengenai fitur tambah data dan edit data - lebih baik menggunakan modal popup atau halaman terpisah?

ANALISIS STRUKTUR DATABASE SIPAKU
=================================

RELASI DATABASE:
mahasiswa (nim) ‚Üê‚Üí rencana_studi ‚Üê‚Üí jadwal ‚Üê‚Üí mata_kuliah (id)
                                      ‚Üì
dosen (nidn) ‚Üê‚Üí jadwal ‚Üê‚Üí ruangan (id)
                ‚Üì
user (kode) ‚Üê‚Üí mahasiswa/dosen

TABEL STRUKTUR:
- mahasiswa: nim (PK), nama, created_at, updated_at
- dosen: nidn (PK), nama, created_at, updated_at  
- mata_kuliah: id_mata_kuliah (PK), kode_mata_kuliah, nama_mata_kuliah, sks
- ruangan: id_ruangan (PK), nama_ruangan
- jadwal: id (PK), nama_kelas, id_mata_kuliah (FK), id_ruangan (FK), nidn (FK), hari, jam
- rencana_studi: id_rencana_studi (PK), nim (FK), id_jadwal (FK), nilai_angka, nilai_huruf (FK)
- user: id_user (PK), nama_user, role, kode
- nilai_mutu: nilai_huruf (PK), nilai_mutu

REKOMENDASI AWAL - HYBRID APPROACH
==================================

MODAL POPUP UNTUK:
‚úÖ Mahasiswa (nim, nama) - 2 fields, complexity 2/10
‚úÖ Dosen (nidn, nama) - 2 fields, complexity 2/10
‚úÖ Mata Kuliah (kode, nama, sks) - 3 fields, complexity 3/10
‚úÖ Ruangan (nama_ruangan) - 1 field, complexity 1/10

HALAMAN TERPISAH UNTUK:
üîÑ Jadwal - 7 fields + complex logic, complexity 8/10
   - Conflict detection (ruangan, dosen, waktu)
   - Real-time availability checking
   - Multiple foreign key relationships

üîÑ Rencana Studi - 4 fields + academic logic, complexity 7/10
   - Academic rules validation
   - GPA calculation
   - Prerequisites checking

ANALISIS KELEBIHAN & KEKURANGAN
===============================

MODAL POPUP:
Kelebihan:
- Faster workflow (tidak kehilangan konteks)
- Quick actions untuk data entry rutin
- Visual continuity
- Mobile friendly
- AJAX integration yang smooth
- Memory efficient
- DataTables integration langsung
- Reusable components
- Faster implementation
- Less files needed

Kekurangan:
- Screen real estate terbatas
- Scrolling issues untuk form panjang
- Validation display terbatas
- Sulit untuk multi-step forms
- Complex validation handling
- File upload limitations
- Focus management challenges
- Accessibility concerns

HALAMAN TERPISAH:
Kelebihan:
- Full screen real estate
- Better information architecture
- Multi-step wizards support
- Rich interactions
- SEO friendly
- Browser navigation works
- Complex validation support
- Better file management
- Separation of concerns
- Easier testing
- Feature expansion friendly

Kekurangan:
- Context loss
- Navigation overhead
- Slower workflow
- Mobile UX bisa heavy
- More routes needed
- More controllers
- More views
- State management complexity
- Page load overhead
- Resource usage tinggi

CONCERN UX CONSISTENCY
======================

MASALAH HYBRID APPROACH:
- Mental model confusion (user expect consistency)
- Workflow disruption (berbeda behavior untuk berbeda form)
- Learning curve issues (harus hafal mana modal, mana halaman)
- Inconsistent button behavior

SOLUSI YANG DIPERTIMBANGKAN:
1. Visual differentiation strategy
2. Progressive disclosure pattern  
3. Contextual explanation strategy
4. Complexity-based differentiation
5. Consistent language/terminology
6. User education approach

KEPUTUSAN FINAL
===============

PILIHAN: UNIFIED MODAL APPROACH
Menggunakan modal untuk SEMUA form dengan ukuran yang disesuaikan:

MODAL SIZES:
- mahasiswa: modal-sm (400px)
- dosen: modal-sm (400px)
- mata_kuliah: modal-md (500px) 
- ruangan: modal-sm (400px)
- jadwal: modal-xl (800px + fullscreen mobile)
- rencana_studi: modal-lg (700px)

ALASAN KEPUTUSAN:
‚úÖ 100% consistent UX
‚úÖ No mental model confusion
‚úÖ Faster workflow
‚úÖ Better mobile experience
‚úÖ Single Page Application feel
‚úÖ Perfect DataTables integration
‚úÖ AJAX-first architecture
‚úÖ Component reusability
‚úÖ Faster implementation
‚úÖ Less code maintenance
‚úÖ Uniform pattern

DISKUSI LAYOUT & CONDITIONAL RENDERING
=====================================
Tanggal: 1 November 2025
Topik: Strategi Layout Architecture untuk Sistem Akademik SIPAKU

OVERVIEW DISKUSI
================
Diskusi mengenai struktur folder layout dan pendekatan terbaik untuk mengelola 
tampilan berbeda berdasarkan role user (Admin, Dosen, Mahasiswa) dalam sistem SIPAKU.

PERTANYAAN UTAMA
================
1. Apakah layout sebaiknya dipisah per role atau digabung?
2. Menggunakan conditional rendering atau file terpisah?
3. Struktur folder yang optimal untuk maintainability?

ANALISIS PENDEKATAN
===================

PENDEKATAN 1: TERPISAH PER ROLE
-------------------------------
Struktur:
app/Views/layout/

‚îÇ   ‚îú‚îÄ‚îÄ main.php
‚îÇ   ‚îú‚îÄ‚îÄ sidebar.php

    ‚îú‚îÄ‚îÄ main.php

‚úÖ Separation of concerns yang jelas

2. TECHNICAL BENEFITS:
   - Simplicity wins untuk project scale ini
   - Faster development dan maintenance
   - Single source of truth untuk navigation
   - Minimal file overhead

3. BUSINESS VALUE:
   - Faster time to market
   - Lower development cost
   - Easier onboarding developer baru
   - Quick iterations dan changes

BEST PRACTICES
==============

1. SECURITY CONSIDERATIONS:
```php
<!-- Helper functions untuk security -->
<?php
function canAccess($role) {
    return session('role') === $role && session('is_logged_in');
}

function hasAnyRole($roles) {
    return in_array(session('role'), $roles);
}
?>

<!-- Usage dengan security check -->
<?php if (canAccess('admin')): ?>
    <li><a href="<?= base_url('admin/mahasiswa') ?>">Mahasiswa</a></li>
<?php endif; ?>
```

2. SCALABILITY PLANNING:
